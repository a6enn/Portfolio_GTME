Model Selection Is an
Architecture Decision.

The question is not "what prompt do I write." It is what type of thinking the task requires, and which model is built for that type of thinking.
Context   Cashbridge build Read   5 min

Three fictional deals. A test bench I built to prove a Clay table could process post-meeting work end to end — meeting summary, conflict detection, CRM log, Slack update, follow-up email. The architecture held up. The columns ran in the right order. Five outputs, clean and usable. With one exception.

The conflict-detection column kept returning a score of zero on a deal where the conflict was obvious. The rep had logged "ready to close end of month." The transcript showed the CFO had moved the meeting twice, outside counsel hadn't been engaged, and the prospect's own estimate was end of Q2. By every reasonable read, that's a stalled deal dressed up as a closing one. The model couldn't see it.

My first instinct was to rewrite the prompt. Tighter framing, more explicit instructions, examples of what counts as a conflict. None of it moved the score. The flag stayed at zero on a deal that anyone who knows enterprise sales would call out in five seconds.

The conflict was real. The model just couldn't find it.

That's when the realization landed. Finding this conflict is not pattern matching. It is inference. You have to read between the lines of a conversation, notice what the rep's framing is leaving out, and draw a conclusion about what the realistic situation actually is. That is a different type of reasoning than summarization or extraction. The default model was good at the things I was asking it to do everywhere else in the table — distill a transcript, write a summary, generate a CRM entry. It was wrong for this column.

I switched models. Same prompt. The flag fired correctly on the same deal.

What I actually learned

I had been treating model selection as an afterthought. Pick the default, write better prompts, debug from there. This build flipped that. The prompt is the instruction. The model is the kind of thinking. They are not the same dimension, and confusing them costs hours.

Looking back across the rest of the table, the columns split cleanly along this line. The meeting summary, the CRM log, the Slack update, the follow-up email — those are all transformation tasks. Take a structured input, produce a structured output in a different shape. A fast, cheap model handles them well. The conflict detector is the only column that has to reason. Read what was said, weigh it against what was claimed, decide whether the gap is meaningful. That is a different cognitive load and it deserves a different model.

The architecture decision in a Clay table — or any AI workflow — is not just which columns do I need and in what order. It is also which type of thinking does each column require, and which model is built for that thinking. Once I started routing columns deliberately, the table got faster, cheaper, and more accurate at the same time. Cheap models on transformation. Strong models only where inference earns its cost.

Where this lives in production

Most AI workflows I see in the wild are using one model everywhere. Sometimes the most expensive one because the team got burned once and overcorrected. Sometimes the cheapest one because nobody wanted to think about routing. Both are leaving real performance on the table.

The discipline is small but it compounds. Before you write the prompt, ask what type of work the column is doing. Extraction. Transformation. Generation. Inference. Judgment. Each of those has a different floor for model capability. Match the floor to the task and the rest of the system gets simpler. Skip the question and you end up either burning credits on tasks that did not need them, or staring at a zero score on a conflict you can see with your own eyes.

The prompt is the instruction. The model is the kind of thinking.

That sentence is the one I keep coming back to. It rewrote how I plan a Clay table, how I size a Claude API workflow, and how I review someone else's prompt when it is not behaving. The build that taught me this looked like a prototype. The principle behind it is the part that scales.
