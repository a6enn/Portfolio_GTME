The Right Skill Architecture Compounds.
The Wrong One Collapses.

Most people treat prompts like instructions. The real leverage shows up when you treat them like programs — and stop building one skill that tries to do everything.

Subject Skill design as program design Read time 5 min

A skill is a saved program. It captures your process, your standards, and your context once — so the model doesn't make you rebuild it from scratch every time.

The leverage isn't the AI getting smarter. It's you getting better at externalizing your thinking into something reusable.

Treat a Prompt Like a Program

A prompt has inputs, logic, and expected outputs. When you write a skill, you're defining a standard for how a type of work gets done. The more precise the skill, the less variation in the output. The less variation, the more you can build on top of it.

I Tried to Do It All in One Skill

My first prospecting skill did everything: ICP scoring, company research, prospect identification, personalized outreach. One skill, one workflow. That's how a manager thinks about a task.

It's not how the best employees execute. A good employee breaks the ask down into smaller, sequenced pieces before doing anything. My monolithic skill was thinking like the manager. It needed to think like the employee.

The output was a mess. Hallucinations in the enrichment. Generic copy where there should have been specificity. The problem wasn't the prompting. It was the architecture.

Different reasoning types → different skills. The orchestrator sequences them.

ICP scoring is pattern recognition. Research is synthesis. Prospect ID is filtering. Outreach is creative generation. Four different types of reasoning, all competing for one context window. The model loses the thread. Hallucinations aren't random — they're what happens when you ask a model to hold too much at once.

One skill should do one type of thinking. Not one task.

Why the Architecture Matters

1. Context windows have walls.

When a monolithic skill runs out of room, it fills gaps by improvising. Smaller skills mean cleaner context, and cleaner context means outputs you can trust.

2. Monolithic skills are brittle to edit.

Update a large prompt and it changes things you didn't intend. A focused skill can be rewritten in isolation. You can improve outreach without risking research logic.

3. Compounding only works on a stable foundation.

Treat your library as a pyramid. The base layer is slow to build, but every layer above it goes faster because you're not starting over. A monolithic system grows sideways. A modular one grows up.

Match the Orchestration to the Scope

If the workflow lives entirely inside Claude, an orchestration skill is enough — one skill that sequences the others and passes context between them. If it crosses into Clay, Salesforce, or LinkedIn, that's where n8n or Zapier earns its keep. Don't over-engineer it.

What the Group Corrected in Me

Inference over trigger words.

I'd been writing skills with explicit trigger phrases — "Run the ICP skill." At five skills, that's manageable. At twenty, it's a system you have to manage on top of the system you're trying to build. Write descriptions well enough that the model picks the right skill from the goal, not the phrasing.

Claude and Clay are not interchangeable.

Claude does reasoning and generation. Clay does enrichment and data at scale. The interesting question is where they hand off. Clay for breadth, Claude for depth, clean handoff between them so neither tool is doing work it wasn't built for. That principle shapes every build I'm running now.

Build a Library Only You Could Have Built

You see a lot of "download my setup" content online. Copy this skill, use this template. It misses the mark — unless uniformity is the goal.

Two people can share a skill and still be completely different operators. What makes yours yours is the way the skills are ordered, what they feed into, what context lives underneath them, what's been refined over months of your work. Borrow ideas. Build something that's yours. The only thing that makes a library powerful is everything it knows about the person who built it.

The world doesn't benefit from everyone running the same monolithic setup downloaded from the same post. It benefits from people building their own libraries.
